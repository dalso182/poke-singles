# Poke-Singles — Claude project notes

Migration of **poke-singles.com**, a Pokémon singles e-commerce store in Costa Rica
(~5,000 SKUs), from **OpenCart 3.0** to a new **Angular + Supabase** stack.

## Stack

- **Angular 21** (standalone components, strict TS, SCSS)
- **Angular Material 21** (Material 3, azure/blue theme, Roboto + Material Icons)
- **Supabase** — DB, REST/Realtime API, auth (planned, not yet wired)
- **SiteGround** hosting (Apache + PHP; **no Node.js** on Shared/Cloud plans)

## Rendering strategy: SPA-first

- App is built as a **client-rendered SPA**. SSR is not configured.
- SiteGround Shared/Cloud plans don't run Node, so true SSR (`@angular/ssr` server
  bundle) is not deployable on the current host. Only Dedicated Servers support Node.
- Static prerendering (SSG) is feasible on SiteGround later — `ng add @angular/ssr`
  + `getPrerenderParams()` querying Supabase, then deploy the `dist/poke-singles/browser/`
  folder with an `.htaccess` SPA fallback. Defer until product-page SEO is a real lever.
- Current Google SERPs for the live OpenCart site rank mostly static info pages
  (payment methods, contact, shipping, categories) — individual card pages are not
  a meaningful traffic source today, so SPA-first is a defensible starting point.

## Local commands

```bash
npm start          # ng serve on http://localhost:4242
npm run build      # default (production) build into dist/poke-singles/browser/
npm run build:dev  # build with --configuration=dev (dev tier)
npm run build:prod # build with --configuration=production
npm run watch      # dev build in watch mode
npm test           # vitest
```

## Deploy to SiteGround (automated SFTP)

```bash
npm run deploy:dev    # ng build --configuration=dev  + SFTP upload to dev.poke-singles.com
npm run deploy:prod   # ng build --configuration=production + SFTP upload to poke-singles.com
```

`scripts/deploy.mjs` reads `.env.local` (gitignored — copy from `.env.local.example`)
for SiteGround SFTP creds and uploads the resulting `dist/poke-singles/browser/`. It
auto-writes the SPA-fallback `.htaccess` at the upload root and supports flags:

- `--only=code|assets|all` (default `all`) — `code` skips the `assets/` subtree, useful
  once the ~5k product images stabilize so iterations don't re-upload them every time.
- `--skip-build` — re-upload an existing `dist/`.
- `--dry-run` — list files without uploading.

Prod uses the unprefixed env vars (`DEPLOY_HOST`, `DEPLOY_USER`, etc.); dev uses the
`DEV_`-prefixed counterparts. Auth is SSH key (preferred) or password.

The script writes `.htaccess` automatically — no manual file management:

```apache
# Auto-generated by scripts/deploy.mjs
RewriteEngine On
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule ^ index.html [L]
```

## Two-tier environment model

| Tier | Frontend | Supabase | Selected by |
|---|---|---|---|
| Local | `npm start` (localhost:4242) | dev project (TBD) | `src/environments/environment.ts` |
| Dev | `dev.poke-singles.com` (TBD) | dev project (TBD) | `ng build --configuration=dev` (no replacement) |
| Prod | `poke-singles.com` | prod project (TBD) | `ng build --configuration=production` (`fileReplacements` → `environment.prod.ts`) |

`src/environments/environment.ts` (dev/local) and `environment.prod.ts` are currently
stubs with empty Supabase URL + anon key. Fill them in once the Supabase projects
are created. `npm run db:types` / `db:push` / `functions:*` scripts in `package.json`
have placeholder `<dev-ref>` / `<prod-ref>` values to swap when the projects exist.

## Supabase scaffolding

`supabase/config.toml` is staged for the CLI; `supabase/migrations/` and
`supabase/functions/` are present but empty. To wire up:

1. Create dev + prod Supabase projects.
2. Replace `<dev-ref>` / `<prod-ref>` placeholders in `package.json` with the project refs.
3. Fill in URL + anon key in both `environment.ts` and `environment.prod.ts`.
4. `npx supabase link --project-ref <dev-ref>` once.
5. Add the Supabase client wiring in `src/app/core/supabase/`.

## Project layout

- `src/app/app.ts` — root component, just renders `<router-outlet />`
- `src/app/app.routes.ts` — top-level routes; Shell wraps all child routes
- `src/app/app.config.ts` — providers: router, async animations, error listeners
- `src/app/shell/` — layout component (mat-toolbar + mat-sidenav-container)
- `src/app/home/` — placeholder home route
- `src/styles.scss` — Material 3 theme (`mat.theme()` with azure-blue palette)

## URL strategy when product pages are built

- The current OpenCart site uses SEO-style URL aliases. **Preserve the same slugs**
  in the new Angular routes (`/products/:slug`) where possible to avoid losing the
  (already-limited) existing Google rankings.
- For any URL pattern that has to change, add 301 redirects via `.htaccess` on
  SiteGround.

## Out of scope right now

These are deliberately deferred and will each get their own plan when picked up:

- Supabase project, client SDK wiring, env vars, RLS policies
- Auth (Supabase Auth — likely email/password + magic link)
- Product / category / set data models, services, pages
- Cart state, persistence, checkout
- Payments: **SINPE Móvil** + bank transfer (matching the existing OpenCart flow)
- OpenCart 3.0 data export → Supabase import (products, categories, customers,
  orders, URL aliases)
- 301 redirect map from old OpenCart URLs
- SSR / static prerendering (`ng add @angular/ssr`)

## Conventions

- Standalone components only; no NgModules.
- Use Angular signals (`signal`, `computed`, `effect`) for component state.
- Material components imported per-component via the `imports:` array.
- Don't access `window` or `document` directly without `isPlatformBrowser()` —
  keeps the door open for adding SSR later without a refactor.
